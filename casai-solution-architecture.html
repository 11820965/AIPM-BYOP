01

Executive summary

The live prototype at casai-intelligent.lovable.app is a high-fidelity, front-end-only clickable prototype. Inspection confirms a React + Vite + TypeScript + Tailwind + shadcn/ui single-page app whose entire state is one localStorage object — {role, name} — with all workers, bookings, scores and alerts hard-coded as seed data in the bundle. There is no backend, no database, and every integration (OTP, Aadhaar eKYC, payments, the five AI engines, notifications) is simulated in the browser.

That is exactly what a prototype should be, and it is a strong one: it proves the product's experience and information architecture across three distinct user contexts. Building Casai for real therefore means keeping the front-end design system and UX, and building everything behind it — identity, services, data, AI, and integrations — while closing the architectural debt the prototype necessarily left open.

The one-line thesis

Casai is not a marketplace with an app bolted on — it is an identity + event platform (verified worker identity, predicted no-shows, GPS check-ins, dual-timezone alerts) with three thin context-specific front-ends over shared services. Get the identity/context boundary and the event/notification spine right, and the rest composes.

What the prototype gets right

  • NRI is the best-architected context — dedicated onboarding, its own dashboard and nav, dual-timezone display, a Home Manager, an alert feed and a weekly summary. (Both source documents call it "not built"; they are stale.)
  • Role-based routing exists — the client already forks Household / Worker / NRI from a single landing page.
  • The trust model is coherent — layered verification (eKYC → police → performance → reliability → community → skill) is the product's genuine differentiator and must survive intact.

Architectural debt to close

Authorization is client-side only

Access context is a string in localStorage.gharseva.state.role. Editing it exposes any context. Production needs a server-issued identity with a role/context claim and server-enforced authorization.

Ops / consumer data boundary leaks

NSM, SLA-breach and churn scores render on the household /app and /app/insights screens; the worker's credit & loan data sits on a household-reachable passport route. These are data-exposure issues, not styling.

Worker context has no separation

/worker renders the household nav — the one context that didn't get its own shell. NRI proves the pattern; worker must adopt it.

No single source of truth

Three pricing schemes in two currencies (₹ services, ₹ subscriptions, $ NRI Care+); the same worker shows 312 / 347 / 412 jobs across screens. Fatal once these numbers drive payments and lending.

Decision — Home Health Score is retired

/app/health 404s and the nav tab is already gone. That ambiguity is now resolved as a deliberate removal, not a bug to fix: the screen, the AI-04 scoring engine, the room-score model and the category service schedules (AC/chimney/geyser) are all out of scope. This is safe because the NRI dashboard grew its own answer to "is the house okay" — the alert feed, wellness check, 7-day summary and monthly heatmap now serve that job for the persona who needed it most. The trade-off to accept knowingly: households lose the upkeep view and its "AI recommends this month" bookable actions, which the PRD called the highest-conversion CTAs in the product. Casai is a staffing-and-trust platform, not a home-maintenance business — cutting this sharpens that focus, and the /app insight cards remain the booking-prompt surface.

Detail — pricing today

Services ₹220/hr cook · ₹180/hr maid · ₹250/hr driver (home tiles' "from ₹199" is a bug). Household subscriptions Lite ₹1,499 / Family ₹3,499 / Care+ ₹5,999. NRI Care+ $49/mo vs Basic $15/mo. The 4th category "Nurse" has migrated into the NRI context as Caregiver (24/7) ₹380/day — a deliberate elder-care reframe worth recording, not a gap.

03

Architecture goals & principles

Context-first

Household, Worker, NRI and Ops are bounded contexts with enforced data boundaries — the organising principle end to end.

API-first & event-driven

Services expose contracts; state changes emit events. AI and notifications react to events, never sit in the request path.

Single source of truth

One catalog/pricing service, one canonical worker record. No screen computes its own numbers.

Mobile-first PWA

India 4G reality: offline-tolerant, installable, <2s loads. Keep the shadcn/Tailwind system from the prototype.

Secure by default

Server-enforced RBAC by context; PII and financial data minimised and scoped; payments tokenised out of PCI scope.

Observable & SLA-bound

Every latency target below is a monitored SLI with an alert, not a hope.

04

System context (C4 · Level 1)

Casai sits between four human actors and a set of Indian identity, messaging, payment and financial systems. The external integrations — all simulated in the prototype — are where most build risk and compliance weight sit.

05

Bounded contexts

The prototype's three route namespaces become four bounded contexts (Ops is implied but unbuilt). Each owns its screens and its read model; they collaborate only through shared services and events — never by reading each other's data directly. This is the fix for both the ops-leak and the worker-nav debt.

06

Container architecture (C4 · Level 2)

One React PWA renders four context shells behind an API gateway / BFF. Nine backend services own their data; an event bus carries state changes to the AI engines, the notification pipeline and the ops read models. This is the smallest decomposition that honours the context boundaries without premature micro-service sprawl — services can start as modules in a modular monolith and split along these seams under load.

07

Backend technology stack

The prototype has no backend whatsoever — no database, no API, no auth server. Network inspection returns only static JS/CSS chunks; every worker, booking, score and alert is a literal in the bundle, and the "session" is a localStorage object. This is the single largest build gap in the product, and the stack below is what closes it.

Two tracks are recommended, in sequence rather than as alternatives: Track A gets the existing prototype a real backend in weeks, and is the natural fit because Lovable integrates Supabase natively and the codebase is already TypeScript. Track B is where the workload migrates as supply, AI and lending mature. The service boundaries in §06 hold across both, so Track A is not throwaway.

Track A — Supabase-native MVP

Fastest credible path from this prototype to a real product. One language end to end, and Postgres Row Level Security enforces the context boundaries in the database — the proper fix for client-side roles.

  • Postgres (managed) — core relational store
  • Supabase Auth — phone + email OTP, JWT with a custom role/context claim
  • RLS policies — household / worker / NRI / ops data boundaries
  • Edge Functions (Deno · TS) — booking logic, webhooks, integrations
  • Realtime — live booking status, NRI alert feed
  • Storage — Aadhaar / PAN / police docs, photos
  • pg_cron + pg_net — the 15-min no-show recompute

Track B — Production scale Scale

Adopted service-by-service when the AI engines, payouts and lending outgrow Track A's edge runtime and pg-based scheduling.

  • Node.js + NestJS (TS) — core services, shares types with the PWA
  • Python + FastAPI — AI engines (ML ecosystem)
  • Postgres + Redis (match cache, hot scores, sessions)
  • Kafka / Pub-Sub — event bus, via transactional outbox
  • Temporal — durable SLA timers & escalation workflows
  • ClickHouse / BigQuery — ops & NSM read models
  • Cloud Run / GKE + Terraform · OpenTelemetry

How RLS fixes the client-side role hole

Today the context is localStorage.gharseva.state.role — editable by anyone. Under Track A the claim is signed into the JWT and the boundary is enforced by the database itself, so no query can cross a context even if the client is tampered with:

-- A household may read only its own bookings
create policy "household reads own bookings" on booking
for select using (
  household_id = auth.uid()
  and (auth.jwt() -> 'app_metadata' ->> 'role') = 'household'
);

-- Churn & NSM are ops-only: no household policy exists at all
create policy "ops reads churn" on churn_score
for select using ((auth.jwt() -> 'app_metadata' ->> 'role') = 'ops');

Two constraints that shape the stack

Data residency — Aadhaar-linked and personal data should stay in an India region (ap-south-1 / asia-south1) under the DPDP Act; pick this at project creation, not later. Dual payment gateways are not optional: households pay ₹ over UPI while NRI Care+ bills $49 internationally, so Razorpay and Stripe must sit behind one Payments service with a single ledger.

08

AI engine layer

The prototype's "AI" is presentation — static strings and fixed percentages. In production four engines sit behind the AI orchestration layer, driven mostly by the event bus so they never block a user request. Two are read-path (matching, voice) and must hit tight latency budgets via caching and streaming.

The PRD specified a fifth — AI-04 Home Health Scoring. It has been retired by product decision (see §02); the engine, its room-score model and its category service schedules are out of scope and are not built.

Placement principle

Scores are precomputed and cached, surfaced as read models — the client never waits on a model. AI-05 (churn) is an ops-only output and must never reach a household read model; that boundary is the fix for the current consumer-facing leak.

09

Data architecture

A relational core (Postgres) holds the transactional entities; Redis caches match results and hot scores; an analytics warehouse holds ops read models. The model below fixes the prototype's integrity problem by making Worker the single owner of stats that today disagree across screens.

Integrity rule

No surface may store or recompute worker stats (jobs, on-time %, credit score). All reads resolve to the Worker service's canonical record. This is a hard precondition for the credit-score and NBFC-loan features — a field that disagrees with itself cannot underwrite a ₹50,000 loan.

10

Key flows

10.1 · Authentication & context resolution

Replaces the client-side role string with a server-issued token carrying a verified context claim — the root fix for the current authorization model.

10.2 · Booking with AI match, confirmation & notification

Fills the prototype's flagged critical gap — a real confirmation screen and a post-booking notification within 60 seconds.

10.3 · No-show prediction → auto-escalation

The pre-emptive loop that makes "we act before problems happen" real: high risk arranges a backup before the no-show, not after.

10.4 · NRI GPS check-in → dual-timezone alert

The highest-priority build and the NRI product's whole reason to exist — a check-in event turned into a <60s notification stamped in both timezones.

Prototype bug to carry into backlog

The live NRI dashboard rendered PST although UK was selected at signup — timezone is not yet bound to the profile. The Notification service must resolve delivery timezone from the linked NRI record, not a hard-coded default.

11

Frontend architecture

Keep what the prototype proved — React + Vite + TypeScript + Tailwind + shadcn/ui + lucide-react, and the whole visual system. Change how context is decided and how data arrives.

Context-aware shell

A single app with four route groups (/app, /worker, /nri, /ops), each with its own nav and layout. The nav is chosen from the server context claim — not a localStorage string — closing the worker-nav and ops-leak gaps together.

Server state, not seed data

Replace bundled mock data with a typed data layer (e.g. TanStack Query) over the BFF. Skeleton loaders and error+retry states become real, per the NFRs.

PWA & offline

Installable, service-worker cached shell, optimistic booking UI — designed for intermittent Indian 4G.

i18n & timezone

Hindi-English code-switch UI; a timezone service renders every timestamp as IST + the viewer's zone, resolved from profile.

Design system preserved

The trust-signal hierarchy and worker-profile layout are the product's differentiator — componentised and locked behind visual-regression tests, changed only via A/B test.

Accessibility built in

44px targets, 4.5:1 contrast, alt text, TalkBack/VoiceOver on the booking-confirmation path — enforced in CI.

12

Cross-cutting concerns

Security & authorization

  • Server-enforced RBAC by context; every read scoped to the caller's claim
  • Ops metrics & churn never in a household read model
  • Worker financial data (credit, loans) worker-scoped only
  • Public passport verify page exposes verification status only

Compliance

  • UIDAI Aadhaar eKYC — licensed AUA/KUA path, consent + audit
  • India DPDP Act — data minimisation, consent, retention
  • Payments tokenised at gateway — keep app out of PCI-DSS scope
  • Consent-based NRI↔household linking (both sides opt in)

Reliability & SLA

  • Replacement-SLA timer is a first-class, monitored process
  • Notification pipeline retries with dead-letter; delivery receipts
  • Graceful degradation — a stub route shows "coming soon", never a 404

Observability

  • Every NFR target is an SLI with alerting
  • North-Star + ops read models from the same event stream
  • No-show precision tracked weekly against actuals

14

Key decisions & risks

Architectural decisions

ADR-01

One PWA, four shells

Single React app, route-grouped by context — not three apps. Shared design system, lower overhead; contexts still isolated by the server claim.

ADR-02

Modular monolith → services

Start services as modules behind one deployable; split along the documented seams under load. Avoids premature micro-service tax.

ADR-03

Event-driven AI

Engines consume the event bus and publish read models; only matching & voice are on the request path. Keeps user latency flat.

ADR-04

Single catalog service

One source for services, plans, prices and currency — every surface reads it. Ends the ₹199/₹220 and ₹/$ divergence.

ADR-05

Managed identity + OTP

Buy, don't build, SMS/email OTP and session management; own only the context-claim logic on top.

ADR-06

Tokenised payments

Gateway-hosted card capture keeps Casai out of PCI-DSS scope; ledger records references only.

ADR-07

Supabase first, migrate by seam

Start on Supabase (§07 Track A) — Lovable-native, TypeScript throughout, RLS enforces context in the database. Migrate service-by-service to Track B along the §06 seams as AI and lending mature.

ADR-08

Dual payment gateways

Razorpay for ₹/UPI households and payouts; Stripe for $ NRI Care+. One Payments service and one ledger in front of both — the currency split is a product fact, not a choice.

ADR-09

Retire Home Health Score

Drop the screen, AI-04 and its service schedules. The NRI dashboard already answers "is the house okay"; Casai is a staffing-and-trust platform, not home maintenance. Accepted cost: the household upkeep view and its booking prompts.

Bottom line

The prototype has de-risked the experience. The build is a backend, data, AI and integration programme behind an already-validated front-end — anchored on two things the prototype couldn't fake: a real identity/context boundary, and an event-driven notification spine.

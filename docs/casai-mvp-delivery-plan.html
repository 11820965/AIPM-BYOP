01

What the MVP is — and isn't

The PRD defines an "MVP Sprint" as seven items: OTP registration, a booking confirmation screen, a date selector, worker pre-selection, moving the NSM dashboard to ops, a Passport tab fix, and pricing consistency. Six of those seven are front-end fixes, and the list quietly assumes a backend exists. None does — no database, no API, no auth server. So the real MVP is considerably larger beneath the surface and should be smaller on it.

MVP definition

One RWA zone in Mumbai. A household finds a verified worker, books, and pays; the worker is dispatched, checks in by GPS, and gets paid. The transaction loop closes end to end with real identity and real money. Everything else — AI, NRI alerts, Academy, loans — is deliberately after. Home Health Score is not deferred but retired.

02

The five rules that set the order

1 · Supply before demand

A household cannot book a verified worker who doesn't exist. Fifty verified workers in the beachhead zone is a hard precondition for any demand-side work — so worker onboarding and eKYC land before the booking loop.

2 · AI comes after data, not before

No-show and churn models train on booking history. At zero bookings there is nothing to learn. Shipping AI in the MVP is not ambitious, it is impossible — the loop must run first and generate the training set.

3 · Calendar-bound work starts on day 1

SMS DLT registration, Aadhaar AUA/KUA licensing and gateway KYC are measured in weeks of waiting, not days of effort. They begin in week 1 regardless of build order, or they become the critical path.

4 · Fork the prototype, don't rewrite it

The Lovable build is real React and its UX is validated. We fork it, strip the seed-data module, and attach a real data layer. Rewriting the UI would cost months and buy nothing.

5 · Walking skeleton, then thicken

A thin end-to-end slice reaches production in week 4 — login, real worker from Postgres, deployed. Integration risk is paid down early instead of discovered in a big-bang merge.

Every phase is demo-able

Each phase ends with a measurable gate and something a stakeholder can use — not a "backend sprint" with nothing to show.

The sequencing argument that matters most

Casai's positioning is "AI home intelligence." That makes it tempting to build the AI first. Rule 2 forbids it. AI-01 (matching) needs reliability histories; AI-02 (no-show) needs worker no-show patterns; AI-05 (churn) needs booking-frequency deltas. The prototype fakes all of this with static numbers, which is exactly why it feels finished and isn't. The MVP's real product is the data exhaust that makes phase 4 possible.

03

The phase plan

P0Foundations & walking skeletonWeeks 1–4

Stand up the spine everything hangs on, and get one thin vertical slice into production. Start every long-lead procurement item on day one.

Ships
  • Supabase project in India region; core schema (Household, Worker, Booking, Subscription)
  • RLS policies per bounded context — the real fix for client-side roles
  • Phone OTP auth; JWT carrying the context claim
  • Fork Lovable FE; delete gharseva.state seed module; typed data layer over Supabase
  • Context-aware shell — /app · /worker · /nri · /ops route groups, nav driven by the server claim
  • Catalog & Pricing service — one source for ₹ and $
  • CI/CD, environments, observability baseline
Exit criteria
  • A real user OTP-logs in on prod and sees a worker served from Postgres
  • Automated test proves a household JWT cannot read ops/churn tables
  • One price renders identically on every surface, from catalog
  • All four procurement tracks formally started

Risk — DLT registration delay blocks SMS OTP. Mitigate: email OTP on staging; DLT filed week 1. Debt closed — client-side auth · ops-metric leak · Passport exposure · worker-nav · pricing split.

P1Supply — verified workers liveWeeks 4–7

Fifty eKYC- and police-verified workers live in the beachhead zone. Until this exists, every demand-side feature is worthless.

Ships
  • The existing 8-step onboarding wizard wired to a real backend
  • Aadhaar eKYC via licensed AUA/KUA aggregator
  • Document upload (PAN, police) → Storage; ops review queue (internal tool, not a custom build)
  • Worker context shell + its own nav
  • Canonical Worker record — single source for jobs / on-time / rating
  • Profile goes live on ops approval; worker notified by SMS
Exit criteria
  • 50 workers eKYC + police verified and live in zone
  • Worker stats identical across every surface (contract test)
  • Ops approves or rejects within 24h

Risk — AUA/KUA licensing slips. Mitigate: manual ops-assisted verification for the first cohort; the wizard already models the wait well. Debt closed — 312/347/412 job-count divergence.

◆ MVP ships at the end of this phase
P2The transaction loopWeeks 7–11

Money moves and the loop closes: book → pay → dispatch → GPS check-in → payout. This is the MVP.

Ships
  • Booking service — create, slots, date selector (14 days), worker pre-selection
  • Booking confirmation screen + booking.confirmed event was critical gap
  • Payments — Razorpay UPI/cards; worker payout ledger (48h SLA)
  • GPS check-in (geofence) → attendance + reliability computation
  • Post-booking push notification <60s (FCM)
  • Post-service review → feeds reliability & skill tags
  • Replacement SLA timer — ops-manual dispatch is acceptable here
Exit criteria — the MVP gate
  • Household books & pays; worker checks in by GPS; worker is paid — real money, end to end
  • Confirmation + push within 60s, measured
  • NSM instrumented; 48h-second-booking guardrail measurable
  • 40 households onboarded in zone

Risk — GPS accuracy in dense Mumbai buildings. Mitigate: tuned geofence radius + manual check-in fallback with ops audit. Note — check-in is built here because reliability and SLA need it, which is what makes phase 3 cheap.

P3NRI care — highest LTVWeeks 11–14

Turn the check-in events P2 already emits into the NRI product. Small increment, largest revenue per user (₹25,164 LTV vs ₹10,800).

Ships
  • NRI onboarding wired — email OTP, international numbers
  • Household ↔ NRI consent linking (code + expiry, both sides opt in)
  • Stripe for $ Care+ subscription; single ledger with Razorpay
  • GPS check-in → dual-timezone alert <60s, timezone resolved from profile
  • No-show alert with replacement ETA; weekly digest
  • /nri/history was stub · Home Manager routing
Exit criteria
  • An NRI in London receives a check-in alert <60s with correct dual timestamps
  • 10 Care+ subscriptions active
  • Razorpay + Stripe reconcile to one ledger

Risk — international notification latency; dual-gateway reconciliation. Debt closed — PST/UK timezone bug · NRI history stub.

P4Make the AI realWeeks 14–18

By now ~3 months of booking history exists — enough to train on. Replace the prototype's static numbers with live engines.

Ships
  • AI-01 Matching — real ranking, P95 <500ms via Redis
  • AI-02 No-show + auto-escalation above 70% risk
  • AI-03 Voice → wired to the real booking engine
  • AI-05 Churn → ops-only read model, never household-facing
Exit criteria
  • No-show prediction precision ≥70% against actuals
  • Matching P95 <500ms
  • Churn score unreachable from any household JWT

Risk — cold start: if P2/P3 volume is thin, models underperform. Mitigate: heuristic rules behind the same API contract, swapped for models when data supports it. Resolved — Home Health Score (AI-04) is retired, not deferred; four engines ship here, not five.

P5Ops & scaleWeek 18+

Run the network at more than one zone, and open the worker economics that make supply self-sustaining.

Ships
  • /ops dashboard — NSM, SLA breach, churn queue, worker map
  • Recurring & multi-worker bookings (path to NSM 4.0)
  • Skill Academy; NBFC loan disbursement
  • Zone expansion — Powai, Thane, Bandra
  • Track B migration, service by service, as load demands
Exit criteria
  • Ops acts on churn/SLA without engineering help
  • Second zone live with 50+ verified workers
  • NSM trending toward 4.0

Risk — premature migration to Track B. Mitigate: migrate only at a proven bottleneck, along the §06 seams.

04

Timeline

Procurement runs in parallel from week 1 — it is the only thing that can silently become the critical path.

Reading the chart

Aadhaar licensing (8 weeks) overlaps P0 and P1 and lands just as P1 needs it — that is deliberate and tight. If it starts even two weeks late it becomes the critical path and pushes the MVP directly. It is the single most schedule-sensitive item in the plan.

If one thing is taken from this plan

File the Aadhaar and DLT paperwork in week 1. They are the only items that can add six weeks to the MVP while every engineer is fully occupied and every sprint looks green.

Bottom line

Eleven weeks to a closed transaction loop in one Mumbai zone; eighteen to NRI plus real AI. The plan's spine is simple: verify supply, close the loop, harvest the data, then make the AI true — in that order, because no other order actually works.

01 The starting point

The brief began with a live Lovable prototype at casai-intelligent.lovable.app. Inspecting it revealed the truth: a front-end-only prototype — React on the surface, but no database, no API, no auth. Every worker, booking and score was hard-coded, the "session" was a localStorage string, and all five "AI engines", eKYC, and payments were faked in the browser.

It looked finished precisely because it was fabricating everything. The work here was to make it real — keep the validated experience, and build the platform underneath it.

03 Architecture at a glance

Stack

  • TanStack Start (React 19, SSR + server fns)
  • Supabase — Postgres + Auth, Mumbai region
  • Tailwind + shadcn/ui · Cloudflare Workers
  • TypeScript end to end

Security model — enforced in the database

  • Row Level Security per bounded context
  • Roles assigned server-side, never client-writable
  • Worker financials hidden from households by construction
  • Ops/churn data unreachable from consumers

The core idea

The prototype kept contexts apart in the UI (a localStorage role). Casai now keeps them apart in Postgres — a tampered client changes nothing, because every read is filtered by RLS using a server-verified role. The UI is never what protects the data.

04 Evidence, not assertion

Every claim was checked against the live database. The same guarantees are pinned by an automated suite that runs in CI on real Postgres (compiled to WASM), where each test maps to a specific defect the prototype had:

  • An anonymous visitor sees nothing; each context sees only its own data
  • A household cannot reach a worker's credit score or the churn score kept about them
  • Role elevation to nri or worker is impossible without the proper server-side path
  • A self-onboarded worker is not bookable until ops verifies them
  • One pricing source — the ₹199-vs-₹220 split that attacked trust at checkout is gone

A genuine bug the tests caught mid-build: a booking trigger, subject to the caller's own permissions, would have rejected every household booking. Fixed and re-verified. That's the value of testing against a real database rather than a mock.

06 Running & sharing it

The backend — Supabase (Postgres + Auth) — runs in the cloud, always on. Its dashboard was only ever opened once, to apply the migrations; that setup is done. You do not open Supabase to run the app. You start the front-end and it reaches Supabase over the internet.

Run it locally

npm installOnce, to fetch dependencies.
npm run devStarts the app on localhost. Open it, then use Guest user or Admin on the landing page.

It needs a .env file with two values — VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY — which point the front-end at Supabase. Both are public by design (the anon key ships in the browser; Row Level Security is what protects the data), so they are safe to share. .env is git-ignored, so a fresh clone won't include it.

Share it with someone

They run the code themselves

Give them the repo plus your two .env values. npm install && npm run dev and they're on the same Supabase — migrations are already applied. (Or they create their own Supabase and run the migrations in supabase/migrations.)

A live link anyone can open

localhost only exists on your machine — no one else can reach it. To hand out a URL, deploy the app (it targets Cloudflare Workers) for a public https://… link that needs no setup from the visitor.
